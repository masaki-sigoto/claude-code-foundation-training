# CLAUDE.md — Claude Code基礎勉強会 教材作成プロジェクト

このファイルは、本プロジェクトでClaude Codeに作業を依頼する際のルールブック（取扱説明書）です。Claude Codeは作業を開始する前に、必ずこのファイルを確認してください。

---

## 1. プロジェクト名

Claude Code基礎勉強会 教材作成プロジェクト

---

## 2. 目的

非エンジニアの業務スタッフ（資料作成・ファイル整理・マニュアル作成・LMS用データ作成・ブログ記事構成作成などにClaude Codeを使う人）を対象とした、勉強会用の教材一式を制作することが本プロジェクトの目的です。

「Claude Codeをなんとなく使っている」状態から、「仕事を任せられる状態」に引き上げる教材を作ることが最終ゴールです。

---

## 3. 対象者の状態

教材の読み手は以下の状態を想定しています。教材制作時も常にこの読み手像を意識してください。

- エンジニアではない。プログラミング経験はない
- Claude Codeは導入済みで、何度か触ったことがある
- チャットのような使い方が中心で、「毎回同じ説明をしている気がする」と感じている
- CLAUDE.md・memory・skills・MCPなどの言葉は聞いたことがある程度で、違いは説明できない
- 業務では資料作成・ファイル整理・マニュアル作成などを日常的に行っている

---

## 4. 教材のゴール

教材を通じて読者が達成すべき状態は次の通りです。

- Claude Codeが「チャットAI」ではなく「手を動かすAIエージェント（自分の代わりに作業してくれるAI）」であることを理解している
- プロジェクトごとにCLAUDE.md（取扱説明書）を用意し、毎回同じ説明を繰り返さない運用ができる
- 作業依頼の基本形（目的・対象・完了条件）を書ける
- 危険な指示（曖昧・範囲無限定・破壊的）を避けられる
- 自分の業務を1つ選んで、小さな手順書（CLAUDE.mdまたはskill案）を書き始められる

---

## 5. 文体ルール

全ての教材ファイルで以下の文体を守ってください。

- **日本語で書く**：英語の直訳調は避ける
- **やさしく書く**：専門用語を並べず、意味を噛み砕く。ただし幼稚にしない
- **たとえ話を使う**：抽象概念は必ず身近な例えに置き換える
- **業務例で説明する**：プログラミング例ではなく、業務スタッフの日常業務（議事録・マニュアル・フォルダ整理など）を題材にする
- **絵文字は使わない**：業務資料として扱うため
- **断定しすぎない**：Claude Codeの仕様は変わる可能性があるため、不明点は「公式情報で確認が必要」と明記する

---

## 6. 説明方針（用語のたとえ話）

以下の用語は、必ずこのたとえ話を添えて説明してください。初出時に一言添え、章内で繰り返し触れるときも必要に応じて短く補足します。

| 用語 | たとえ話 |
|---|---|
| Claude Code | 手を動かして作業してくれるAIエージェント（自分の隣で実際にファイルを開いたり直したりしてくれるAI） |
| CLAUDE.md | プロジェクトの取扱説明書（新しく入ったアルバイトに最初に渡すマニュアル） |
| memory | 作業しながら覚えた注意点のメモ（付箋に書き留めたリマインダー） |
| skills | 繰り返し作業の専用レシピ（決まった手順書をファイル化したもの） |
| MCP | 外部ツールとの接続口（他のアプリにつなぐためのコンセント） |
| hooks | 決まったタイミングで自動実行するスイッチ（特定の動作に連動して動くセンサー） |
| subagents | 専門作業を任せる別担当AI（特定分野を得意とする別のスタッフ） |

これらの用語は、業務スタッフが直感的にイメージできるように、初出時は必ずたとえ話と一緒に紹介してください。

---

## 7. 禁止事項

以下は本プロジェクトの教材で「やってはいけない」ことです。

- **危険操作の推奨**：「フォルダごと削除」「ファイルを一括で書き換え」など、取り返しのつかない操作を初心者に推奨する書き方はしない
- **大量編集の推奨**：「100ファイルまとめて処理してもらいましょう」のような、範囲の広すぎる依頼を奨励しない。まずは1ファイルずつを基本とする
- **MCP／hooks必須扱い**：MCP・hooks・subagentsを「最初から必ず使うべき機能」として書かない。これらは概念理解までに留め、CLAUDE.mdとskillsの使い分けを重点的に扱う
- **専門用語の放置**：「エージェント」「プロンプト」「コンテキスト」などの専門用語を、意味の説明なしに使わない
- **万能ツール表現**：「Claude Codeがあれば何でもできる」「全自動」「完璧」といった過度な表現はしない
- **AI出力の無確認**：「Claude Codeに任せておけばOK」とは書かない。必ず「最終的な確認は人が行う」というスタンスを維持する

---

## 8. 成果物の保存方針

成果物は以下のフォルダ構成に従って保存してください。新しいファイルを作る際は、用途に合った既存フォルダに配置し、該当するフォルダがない場合のみ相談の上で新設してください。

```
ClaudeCode基礎勉強会/
├── README.md
├── CLAUDE.md
├── 01_curriculum/      # 勉強会の設計
├── 02_slides/          # スライド構成と本文
├── 03_scripts/         # 講師台本
├── 04_handouts/        # 受講者ハンドアウト
├── 05_exercises/       # 演習課題
├── 06_templates/       # 各種ひな形
├── 07_skills/          # スキル説明とサンプル
├── 08_mcp/             # MCP説明資料
├── 09_faq/             # よくある質問
└── 10_cheatsheets/     # チートシート
```

ファイル名は日本語を避け、英小文字とアンダースコアで命名します（例：`curriculum_overview.md`）。

---

## 9. 作業時の報告ルール

Claude Codeが作業を完了したら、以下を必ずセットで報告してください。

1. **作成・編集したファイルの一覧**：絶対パスで列挙する
2. **各ファイルの概要**：1〜2行で「このファイルに何を書いたか」
3. **不足していると思われる点**：読み手視点で「ここはまだ弱いかも」と感じた箇所
4. **次に着手すべき改善点**：次回の作業提案（優先度付き）

この4点セットを守ることで、人間側のレビューが効率化されます。

---

## 10. 初心者向け表現ルール

読み手が初心者であることを常に意識してください。

- **専門用語は必ず噛み砕く**：初出時に「〜とは〜のこと」を添える
- **抽象語の前に具体例**：「効率化」ではなく「30分の作業が5分になる」と書く
- **画面操作は順序立てて書く**：「まず〜、次に〜、最後に〜」の形
- **結論を先に書く**：「これは何のための話か」を最初の1〜2行で示す
- **長い文章は箇条書きに**：3つ以上の要素が並ぶ場合は箇条書きにする
- **2〜3文で1段落**：段落を長くしすぎない

---

## 11. たとえ話の活用方針

抽象的な説明の直後には、必ず「つまり」「たとえば」で始まる一言を添えてください。

悪い例：
> CLAUDE.mdはプロジェクトのコンテキストを定義するファイルです。

良い例：
> CLAUDE.mdは、プロジェクトごとの取扱説明書にあたるファイルです。たとえば、新しくアルバイトが入ったときに「このお店ではこう呼んでいます」「この資料はここに保存してください」と最初に渡すマニュアルをイメージしてください。

業務スタッフの日常で出てくる場面（議事録・マニュアル整備・フォルダ命名・報告書作成など）を題材に選ぶと、読み手の理解が早まります。

---

## 12. 確認を促すスタンス

全ての教材で、最終的な判断は人間が行うという前提を崩さないでください。

- 「Claude Codeの出力は必ず目を通してから使う」
- 「仕様は変わる可能性があるので、不明点は公式情報で確認する」
- 「本番データへの影響がある操作は、事前にバックアップを取る」

この3点は、各教材の関連箇所で繰り返し触れる方針です。
