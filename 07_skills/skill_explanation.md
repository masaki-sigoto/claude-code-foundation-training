# skills 解説資料（業務スタッフ向け）

この資料は、Claude Code の「skills」という仕組みを、非エンジニアの業務スタッフが理解し、使いこなせるようになることを目的としています。

---

## 1. skills とは何か

skills とは、**繰り返し発生する特定の作業に対する「専用レシピ」**です。

たとえ話で説明します。

- あなたが料理人だとして、毎日いろいろな料理を作るとします。
- 「いつも使う基本の塩加減・火加減」は頭に入っているので、毎回説明されなくてもできます（これが CLAUDE.md 的な「常識」）。
- 一方、「本格ボルシチの作り方」は特別な時しか作らないので、そのときだけレシピ本を開きます（これが skills 的な「専用レシピ」）。

skills も同じで、**必要なときだけ Claude が開いて見る「専用レシピのページ」**です。
ページ 1 枚 = 1 つのスキル = 1 つの作業、と覚えるとシンプルです。

---

## 2. なぜ skills が必要なのか

「全部 CLAUDE.md に書けばいいのでは？」と思うかもしれません。
しかし、CLAUDE.md にすべてを書き込んでいくと、次のような問題が起きます。

- **CLAUDE.md がどんどん長くなる**
  すべての作業手順、すべての注意事項、すべてのテンプレートを 1 ファイルに書くと、数百〜数千行の巨大ファイルになります。
- **今の作業に関係ない情報まで毎回読み込まれる**
  営業メールを書くときに「LMS のビルド確認手順」が混ざっていても邪魔なだけです。
- **人間にとっても読みにくい**
  メンテナンスする人（あなた）も、どこに何が書いてあるのか分からなくなります。
- **Claude の集中を削ぐ**
  関係ない指示が視界に入ると、Claude の判断もブレやすくなります。

そこで、**「いつも必要な情報は CLAUDE.md」「特定作業のときだけ必要な手順は skills」**と分けるのです。

---

## 3. skills と CLAUDE.md の使い分け

| 観点 | CLAUDE.md | skills |
| --- | --- | --- |
| 役割 | プロジェクト全体の常識・ルール | 特定作業の専用手順 |
| 読まれるタイミング | 毎回の会話で常に参照される | その作業が必要になったときだけ |
| 書く内容 | コミットルール、口調、禁止事項、共通の前提 | 営業メールの書き方、議事録のタスク化手順など |
| 長さ | できるだけ短く | 1 スキル 1 ファイル、必要な範囲に絞る |
| 数 | 原則 1 つ | いくつ作ってもよい（作りすぎ注意） |

**使い分けの判断基準：**
- 「この情報、全作業で必要？」→ YES なら CLAUDE.md
- 「この情報、特定の作業のときだけ必要？」→ skills
- 「この情報、1 回限り？」→ そもそも書かず、チャットで伝える

---

## 4. skills と subagents の違い

skills と subagents はどちらも「特定作業のための仕組み」ですが、役割が違います。

| 観点 | skills | subagents |
| --- | --- | --- |
| 実体 | 手順書（Markdown ファイル） | 別の Claude インスタンス（実行者） |
| 役割 | 「どうやるか」のレシピ | 「誰がやるか」の担当者 |
| コンテキスト | メインエージェントが読む | 別のコンテキストで独立して動く |
| 使いどころ | 手順を統一したい／テンプレ化したい | 作業を分担したい／コンテキストを節約したい |
| 組み合わせ | subagent が skill を読むこともある | skill の中から subagent を呼ぶこともある |

簡単に言うと、**skill は「レシピ本」、subagent は「料理人」**です。
料理人にレシピを渡して作ってもらう、というイメージで組み合わせて使えます。

---

## 5. skill が呼び出される仕組み（概念）

skill は、必要になったときだけ読み込まれます。仕組みの概念はこうです。

1. ユーザーが Claude に依頼する（例：「営業メールのドラフトを作って」）
2. Claude は利用可能な skills の「名前」と「説明（description）」を見る
3. 今回の依頼に合いそうな skill があれば、その SKILL.md の本文を読み込む
4. そこに書かれた手順に沿って作業する

つまり、**skill の description（説明文）がマッチングの決め手**になります。
「どんなときに使うスキルか」を description に明確に書いておくことが大切です。

> **注記**: 実際のマッチング動作は Claude Code のバージョンや設定によって変わる可能性があります。正確な仕様は公式情報（公式ドキュメント）での確認を推奨します。

## 6. SKILL.md の基本構造

skill は 1 つのフォルダに `SKILL.md` という 1 枚のファイルを置くのが基本形です。

### frontmatter（ファイル冒頭の設定欄）

```markdown
---
name: sales-email
description: 営業メール（初回アプローチ／フォローアップ／失注後）を作成するときに使う
---
```

- `name`：スキルの識別名。kebab-case（ハイフン区切り）で書く
- `description`：1〜2 行で「いつ使うスキルか」を書く。これがマッチングの判断材料になる
- その他のフィールドが追加される可能性もあるため、**最新仕様は公式ドキュメントで確認**してください

### 本文

frontmatter の下には、以下のような内容を書きます。

- このスキルが何をするか
- いつ使うか／使ってはいけない場面
- 事前に準備するもの
- 具体的な手順（1, 2, 3, ...）
- 成果物の形式
- 品質チェック観点
- 使用例（入力例・出力例）

本資料の「ファイル 2〜6」のサンプルが、この構造の実例になっています。

## 7. 良い skill・悪い skill の例

### 良い skill の特徴

- **1 つの明確な作業に特化**している（例：「議事録からタスクを抽出する」）
- **入力と出力が決まっている**（議事録テキスト → タスク一覧表）
- **手順が具体的**（「まず発言者別に分類、次に担当者を特定、…」）
- **使うべき場面・使ってはいけない場面**がはっきりしている

### 悪い skill の特徴

- **あいまい**（例：「いい感じにドキュメントを整える」）
- **対象範囲が広すぎる**（例：「マーケティング全般」）
- **CLAUDE.md に書けば済む内容**（例：「常にですます調で書く」）
- **手順がなく、精神論だけ**（例：「ユーザー目線で考える」）

## 8. skill 設計のコツ

skill を作るときは、次の順番で進めるとうまくいきます。

1. **まず自分が手作業でやっている手順を書き出す**
   紙でもメモ帳でも OK。普段の自分の動きを分解する。
2. **それを SKILL.md のフォーマットに整える**
   「いつ使うか」「手順」「成果物」「チェック観点」に整理。
3. **実際に使ってみて調整する**
   Claude にやらせてみて、うまくいかない部分を手順に追記する。
4. **3 回くらい使ってから完成とする**
   1 回目で完璧な skill は作れません。運用しながら育てる前提で。

## 9. フォルダ構成の推奨

skills は次のような構成にすると管理しやすいです。

```
<プロジェクトルート>/
├── CLAUDE.md                  ← プロジェクト全体のルール
└── .claude/
    └── skills/
        ├── sales-email/
        │   └── SKILL.md
        ├── meeting-to-tasks/
        │   └── SKILL.md
        └── blog-structure/
            └── SKILL.md
```

- 1 つの skill = 1 つのフォルダ = 1 つの `SKILL.md`
- フォルダ名と `name` は一致させると迷わない
- skill に付随する補助ファイル（テンプレート、サンプルなど）も同じフォルダに入れる

> **注記**: skills を配置するディレクトリの公式な場所や命名規則は、Claude Code のバージョンによって異なる場合があります。最新の配置ルールは公式ドキュメントでご確認ください。

## 10. よくある失敗

- **skill を作りすぎる**
  とりあえず作っておこう、で量産すると、どれを使うべきか Claude も人間も迷います。
- **似た skill が乱立する**
  「メール作成」「営業メール」「顧客メール」が別々にあると混乱します。統合しましょう。
- **更新されずに古くなる**
  業務フローが変わったのに skill がそのままだと、古い手順を繰り返してしまいます。
  月 1 回でも「このスキルまだ使える？」の棚卸しを。
- **CLAUDE.md との重複**
  CLAUDE.md に書いたことを skill にも書くと、矛盾が起きたときにどちらが正しいか分からなくなります。

## 11. 初心者の最初の skill としておすすめのテーマ

最初の skill は、**「あなたが週に 1 回以上やっている、定型的な作業」**から選びましょう。

- 営業メール・フォローアップメール
- 議事録からタスク抽出
- 週次レポートの作成
- ブログ記事の構成案づくり
- 問い合わせメールへの返信テンプレ

「毎回ゼロから考えているけど、実は同じパターン」という作業こそ、skill 化の最適解です。

## 12. 次のステップ

本フォルダの `sample_skills/` に、5 つのサンプル skill を用意しています。

- `sales-email/` 営業メール作成
- `lms-build-check/` LMS 投入前データチェック
- `blog-structure/` ブログ記事構成作成
- `meeting-to-tasks/` 議事録からタスク抽出
- `folder-organizer/` フォルダ整理提案

まずはこれらを読んで、「どういう粒度で」「どういう書き方で」書かれているかを感じ取ってください。
その後、自分の業務に合わせて 1 つ目の skill を作ってみましょう。
