# Claude Code基礎勉強会 演習集

本演習集は、非エンジニアの業務スタッフが Claude Code を業務で使いこなせるようになることを目的としています。
演習1〜5は、受講者が自分の業務を起点にして、CLAUDE.md・skill・指示文・計画確認を体験できる流れになっています。

> たとえ話
> - CLAUDE.md = 取扱説明書（このプロジェクトのルールブック）
> - memory = 作業中の注意点メモ（今回の仕事で気をつけること）
> - skills = 専用レシピ（繰り返し使う手順書）
>
> Claude Codeの仕様は更新されることがあります。本文中の具体的なファイル名・格納場所などで不明な点は、必ず公式情報で確認してください。

## 演習1：自分の業務をClaude Codeに説明する

- 所要時間：15分（書き出し5分＋共有10分）
- 難易度：初級

### 目的
Claude Code に「自分が普段どんな業務をしているのか」を伝えるための材料を、5分で書き出せるようになる。
AIに仕事を任せる前提として「前提条件の共有」がなぜ重要かを体感する。

### 背景（なぜこの練習が必要か）
業務スタッフが Claude Code にうまく仕事をさせられない最大の原因は、
「AIが自分の業務を知らない前提で指示を出していない」ことです。
逆に言えば、最初に前提情報を渡せるだけで、Claude Code の出力品質は大きく変わります。
この演習は、その「前提情報の書き出し」を習慣化するための最初のステップです。

### 設問
次の5項目について、自分の業務を1つ選び、各項目を1〜3行程度で書き出してください。
所要時間は5分を目安にします。

1. 業務の前提（何の仕事か、どの顧客・社内向けか）
2. 登場人物（誰が関わるか。上司、取引先、他部署など）
3. 使うフォルダ／ファイル（どこに資料があるか）
4. 目的（その業務のゴールは何か）
5. 禁止事項（やってはいけないこと、触ってはいけないもの）

### 取り組み方（手順）
1. 紙やテキストエディタを用意する
2. タイマーを5分にセットする
3. 上記5項目を順に書き出す。完璧でなくてよい
4. 書き終わったら、隣の人と交換して「これだけ読めば自分の業務を想像できるか」を確認し合う

### 提出物／成果物
- 上記5項目を箇条書きで埋めたメモ（テキストファイルでも紙でも可）

### 採点観点（自己チェックリスト）
- [ ] 業務名が一言で分かるか
- [ ] 関係者が具体的な役職・部署名で書かれているか
- [ ] フォルダは「どこにあるか」まで書かれているか（社内ドライブ、ローカルなど）
- [ ] 目的が「何を誰に届けるか」まで書かれているか
- [ ] 禁止事項が1つ以上書かれているか

### 良い回答例
```
1. 業務の前提：毎月の月次営業レポートを経営会議向けに作成する
2. 登場人物：営業部の担当者（データ提供）、経営企画室（提出先）、上司（確認）
3. 使うフォルダ：共有ドライブ内の「01_月次レポート/2026年度/」配下
4. 目的：経営会議で使える、前月比の売上・受注件数をまとめたPDFを作ること
5. 禁止事項：元データ（CSV）の上書き保存、未確定の数値を確定値として出すこと
```

### 悪い回答例（よくある失敗）
```
1. いつものレポート
2. うちの部署の人
3. ドライブのどこか
4. 提出すること
5. とくになし
```
- 「いつもの」「どこか」では Claude Code に伝わりません
- 「とくになし」は要注意。禁止事項は必ず1つは考える

### 講師向けガイド
- 5分は厳守。考え込ませるより「書いてみる」が目的
- 巡回で見るポイント：
  - 固有名詞（部署名、ファイル名）が入っているか
  - 「完璧に書こう」として手が止まっていないか
- 書けない人には「一番最近やった仕事を思い出してください」と声をかける
- 共有フェーズでは「隣の人の業務を想像できたか」を必ず聞く

## 演習2：CLAUDE.mdを作ってみる

- 所要時間：25分
- 難易度：初級

### 目的
演習1で書き出した内容を元に、CLAUDE.md の雛形を実際に埋めて「自分専用の取扱説明書」を作る。

### 背景
CLAUDE.md は、プロジェクトごとに Claude Code に読ませる「取扱説明書」です。
毎回同じ前提を口頭で伝えるのは非効率なので、ルールをファイルにまとめておくことで、
Claude Code が自動的にそれを参照してくれる、という発想です。
（※具体的な参照のされ方は環境によって異なるため、公式情報の確認が必要です）

### 設問
`06_templates/claude_md_template.md` の雛形をコピーし、演習1で書き出した内容を元に全セクションを埋めてください。
空欄のまま残すセクションがあっても構いませんが、その場合は「未定」と明記してください。

### 取り組み方（手順）
1. `06_templates/claude_md_template.md` を開き、内容を自分の作業フォルダにコピーする
2. ファイル名を `CLAUDE.md` にする
3. 上から順にセクションを埋めていく
4. 書き終わったら、「この CLAUDE.md だけを読んだ他人が、自分の業務を代行できそうか」を自問する

### 提出物／成果物
- 自分用の `CLAUDE.md` ファイル（最低限、目的・関係者・使用フォルダ・禁止事項の4つは埋まっていること）

### 採点観点（自己チェックリスト）
- [ ] プロジェクト名と目的が1〜2行で書かれている
- [ ] 使用する主要フォルダが具体的なパスで書かれている
- [ ] 絶対に触ってはいけないファイル／フォルダが明記されている
- [ ] 生成物の保存先ルールが決まっている
- [ ] 報告形式（完了時・失敗時）が書かれている

### 良い回答例
```
# CLAUDE.md

## プロジェクト名
月次営業レポート自動化

## 目的
毎月の経営会議向け営業レポートの作成時間を半減する

## 使用する主要フォルダ
- 共有ドライブ/01_月次レポート/2026年度/
  - input/ : 営業部から届く元データ（CSV）
  - output/ : 生成したレポート（PDF・Excel）
  - archive/ : 先月以前のレポート保管

## 絶対に削除・変更してはいけないファイル
- input/ 配下の元データCSV（上書き厳禁。必ずコピーして扱う）
```

### 悪い回答例
```
# CLAUDE.md
いい感じにやってください。
```
- セクション分けがない／指示が曖昧／「いい感じ」は禁句

### 講師向けガイド
- 最初の5分は全員でテンプレートをコピーさせ、スタートラインを揃える
- 巡回で見るポイント：
  - セクションを飛ばしていないか
  - 「どこに保存するか」が書けているか
  - 「禁止事項」に具体が入っているか
- 完璧を求めず「今日書けるところまで」でよいと伝える

## 演習3：悪い指示を良い指示に直す

- 所要時間：20分
- 難易度：初級

### 目的
曖昧な日本語指示が、なぜ Claude Code に伝わらないのかを体感し、
「目的・対象・制約・成果物形式・確認ステップ」を入れる書き方を身につける。

### 背景
Claude Code は言われた通りに動きますが、人間同士の「察し」は働きません。
「資料作っといて」と言われて完璧に察してくれる部下はいないのと同じです。
よい指示には、最低限次の5要素が必要です。
1. 目的（何のためにやるか）
2. 対象（どのファイル・フォルダか）
3. 制約（やってはいけないこと、条件）
4. 成果物の形式（何をどう出してほしいか）
5. 確認ステップ（途中で見せてもらうか、完了後に確認するか）

### 設問
次の5つの「悪い指示」を、上記5要素が含まれる「良い指示」に書き直してください。

1. 「資料作っといて」
2. 「このフォルダ整理して」
3. 「いい感じにメール書いて」
4. 「エラー直して」
5. 「ブログ書いて」

### 取り組み方（手順）
1. まず「なぜこの指示はダメなのか」を1行でメモする
2. 5要素を意識しながら書き直す
3. 書き直した指示を、隣の人に見せて「これで動けるか」を確認する

### 提出物／成果物
- Before/After 形式で5組書き直したテキスト

### 採点観点（自己チェックリスト）
- [ ] 目的が書かれているか
- [ ] 対象が具体的に書かれているか（ファイル名・フォルダパスなど）
- [ ] 制約・禁止事項が1つ以上あるか
- [ ] 成果物の形式（拡張子、分量、構成）が指定されているか
- [ ] 「作業前に計画だけ見せて」など確認ステップがあるか

### 良い回答例

#### 1. 「資料作っといて」
```
Before: 資料作っといて
After:
【目的】来週の営業会議で使う、先月の受注状況サマリーを作ってほしい
【対象】共有ドライブ/01_月次レポート/2026_03/input/sales_data.csv
【制約】元データは絶対に上書きしない。数値は千円単位で丸める
【成果物】A4・1ページのPDF。構成は「総受注額」「前月比」「要注意案件3件」
【確認】作業前に、どの数値を拾うかの計画だけ先に見せてください
```

#### 2. 「このフォルダ整理して」
```
Before: このフォルダ整理して
After:
【目的】デスクトップの「仮置き」フォルダを、来週から使える状態にする
【対象】~/Desktop/仮置き/ 配下（サブフォルダ含む）
【制約】PDFとExcelは消さない。画像は削除してよい
【成果物】整理後のフォルダ構成図（テキスト）と、移動したファイルのリスト
【確認】削除・移動の前に、対象ファイル一覧を必ず提示してください
```

#### 3. 「いい感じにメール書いて」
```
Before: いい感じにメール書いて
After:
【目的】取引先A社の田中様宛てに、来週の打ち合わせ日時変更の連絡メールを送る
【対象】宛先：田中様／件名：打ち合わせ日程変更のお願い
【制約】敬語は「ですます」。絵文字・ビックリマーク不可。200字以内
【成果物】件名と本文のテキスト（署名は入れなくてよい）
【確認】送信はしないでください。本文案だけ提示してください
```

#### 4. 「エラー直して」
```
Before: エラー直して
After:
【目的】昨日から動かなくなった月次集計マクロを直したい
【対象】共有ドライブ/ツール/月次集計.xlsm／エラー文言「実行時エラー 1004」
【制約】元ファイルは変更せず、コピーを作って修正すること
【成果物】修正版ファイルと、どこを何のために直したかの1ページ説明
【確認】修正前に、原因の仮説を3つ挙げて、どれから試すか計画を見せてください
```

#### 5. 「ブログ書いて」
```
Before: ブログ書いて
After:
【目的】自社サイトの採用ブログ向けに「新入社員1か月の振り返り」記事を作る
【対象】素材：社内インタビュー議事録（~/Documents/interview_0415.txt）
【制約】固有名詞は伏せる。である調ではなく「ですます」。1200〜1500字
【成果物】Markdown形式。タイトル案3つ・本文・見出し4つ以上
【確認】本文を書く前に、見出し構成だけ先に提示してください
```

### 悪い回答例（よくある失敗）
```
After: 資料をちゃんと作ってください。よろしくお願いします。
```
- 「ちゃんと」は主観で、Claude Code に伝わらない
- 「対象」「形式」「確認ステップ」が抜けている

### 講師向けガイド
- 1問ずつ順番にやると時間が足りないので、5問一気に取り組ませる
- 巡回で見るポイント：
  - 5要素のうち抜けがちなのは「制約」と「確認ステップ」
  - 「敬語で」など曖昧な制約は、具体化を促す（例：「ですます調」）
- 最後に、参加者から1人ずつ「一番うまく直せた1問」を発表させると学びが深まる

## 演習4：簡単なskill案を作る

- 所要時間：25分
- 難易度：中級

### 目的
自分が毎週・毎月繰り返している作業を1つ挙げ、SKILL.md（専用レシピ）の骨組みを書けるようになる。

### 背景
skill は「繰り返し使う手順書」です。料理で例えれば、毎回レシピを一から書くのではなく、
一度作ったレシピカードを使い回すイメージです。
自分の繰り返し業務を skill 化しておくと、毎回同じ説明を Claude Code にしなくて済みます。
（※Claude Code における skill の具体的な仕組み・ファイル配置は公式情報の確認が必要です）

### 設問
自分が週1回以上繰り返している業務を1つ選び、次の骨組みに従って SKILL.md の下書きを書いてください。

- スキル名（ひと言で）
- 使う場面（いつ呼び出すか）
- 前提条件（始める前に必要な情報・ファイル）
- 手順（番号付きで5〜10ステップ）
- 成果物（何ができあがるか）
- 注意事項（失敗例・よくあるつまずき）

### 取り組み方（手順）
1. 繰り返している業務を3つ書き出し、その中から1つ選ぶ
2. 上記の骨組みを埋めていく
3. 書き終わったら、自分でその手順を読み上げて「これで他人が再現できるか」を確認

### 提出物／成果物
- SKILL.md の下書き（1ファイル）

### 採点観点（自己チェックリスト）
- [ ] スキル名が動詞＋目的語で書かれている（例：「月次レポートを作る」）
- [ ] 使う場面が「いつ・どの状況で」で書かれている
- [ ] 手順が番号付きで、1ステップ1動作になっている
- [ ] 成果物の形式が明記されている
- [ ] 注意事項（失敗例）が1つ以上ある

### 良い回答例
```
# SKILL: 月次営業レポートを作成する

## 使う場面
毎月1日、前月の営業データが揃ったタイミングで実行する

## 前提条件
- 共有ドライブ/01_月次レポート/{年}/{月}/input/sales_data.csv が存在すること
- 前月分のテンプレート（report_template.xlsx）がテンプレートフォルダにあること

## 手順
1. input/sales_data.csv を読み込む
2. 金額を千円単位に丸める
3. 前月比を計算し、5%以上の増減を赤字・青字で強調
4. 上位3社の受注金額を抽出
5. テンプレートにはめ込み、PDFとして output/ に保存
6. ファイル名は「月次レポート_YYYYMM.pdf」

## 成果物
- output/月次レポート_YYYYMM.pdf
- 作業ログ（どのデータを使ったか）

## 注意事項
- 元CSVは絶対に上書きしない（コピーして扱う）
- 金額が空欄の行はスキップし、ログに記録
```

### 悪い回答例
```
# SKILL: いい感じにレポート作る

## 手順
1. やる
2. 提出する
```
- スキル名が抽象的／手順が粗すぎる／成果物の形式が不明

### 講師向けガイド
- 「繰り返しの業務が思いつかない」人には、「先週やった仕事」「先月やった仕事」を聞いて、共通するものを探させる
- 巡回で見るポイント：
  - 手順が「1. やる」のように粗くなっていないか
  - 成果物の形式（拡張子・保存先）が書かれているか
- 時間が余ったら、skill の「使う場面」を具体的に書き直す時間を取る

## 演習5：作業前計画を出させる

- 所要時間：20分
- 難易度：中級

### 目的
Claude Code に「いきなり作業を始めず、先に計画だけ出してください」と依頼する練習をし、
事前確認の習慣を身につける。

### 背景
Claude Code は指示されるとすぐ作業を始めてしまいます。
しかし、意図と違う方向に進んでから止めるより、作業前に計画を確認する方が圧倒的に速くて安全です。
この習慣があるかないかで、やり直しの回数と修正工数が大きく変わります。

### 設問
次の3つのシナリオから1つ選び、「作業は始めず、計画だけ先に提示してください」と依頼する指示文を書いてください。
そして、可能であれば実際に Claude Code に投げて、返ってきた計画をレビューしてみてください。

#### シナリオ
A. 自分のデスクトップを整理したい
B. 過去1か月の議事録から、ToDo だけを抽出したい
C. 新しく受け取ったExcelファイルを、別フォーマットに変換したい

### 取り組み方（手順）
1. シナリオを1つ選ぶ
2. 演習3で学んだ5要素（目的・対象・制約・成果物・確認）を含めた依頼文を書く
3. 依頼文の最後に「**作業は始めないでください。まず計画だけ提示してください**」を入れる
4. （可能なら）実際に Claude Code に投げる
5. 返ってきた計画を読み、以下の3点を確認：
   - 自分の意図と合っているか
   - 抜けている工程はないか
   - 危険な操作（削除・上書き）が含まれていないか
6. 修正点があれば、再度依頼文を調整して投げ直す

### 提出物／成果物
- 依頼文（テキスト）
- （実行した場合）Claude Code から返ってきた計画と、それに対する自分のレビューコメント

### 採点観点（自己チェックリスト）
- [ ] 「作業は始めず計画だけ」と明記しているか
- [ ] 計画に含めてほしい項目（手順・所要時間・リスク）を指定しているか
- [ ] 制約・禁止事項が書かれているか
- [ ] 返ってきた計画を読んで、1つ以上の修正点を指摘できたか
- [ ] 意図と合っている／合っていないの判断理由を言語化できたか

### 良い回答例
```
【目的】デスクトップの「仮置き」フォルダを、来週の作業開始前に整理したい
【対象】~/Desktop/仮置き/ 配下
【制約】
- PDF・Excelは絶対に削除しない
- スクリーンショット画像（png）は削除可
- 判断に迷うものは「要確認」フォルダに退避
【成果物】整理後のフォルダ構成図と、移動・削除ファイルのリスト

【重要】
作業はまだ始めないでください。
まず、以下を含む「作業計画」だけを提示してください：
1. これから実行する手順（番号付き）
2. 削除・移動対象になりそうなファイル数の見込み
3. 想定されるリスクと、その回避策
4. 所要時間の目安
私が計画を確認してOKを出してから、実作業に進んでください。
```

### 悪い回答例
```
デスクトップ整理して。あ、計画も出しといて。
```
- 「計画も」では、作業と同時並行で進んでしまうリスクがある
- 「先に計画だけ」「確認後に作業開始」を明示していない

### 講師向けガイド
- この演習は「実行前に止まる」ことが目的。実際に投げられない環境でも、依頼文の完成度で評価できる
- 巡回で見るポイント：
  - 「計画だけ先に」と明示されているか
  - 計画に含めてほしい項目が具体的に指示されているか
- 実行させる場合は、時間を区切り、計画が返ってきた時点で一度全体共有するとよい
- 最後に「計画を確認する習慣があると、やり直しが減る」ことを体感として言語化させる
