## 共通の書き方のコツ

- {ここに書く} は必ず具体的に埋める。「いい感じに」「なるはや」は禁止
- 対象ファイル・フォルダは絶対パスで書くのが望ましい
- 「作業前に計画だけ先に見せてください」を末尾に入れる癖をつける
- 迷ったら、一度に全部を任せず、段階的に依頼する

## テンプレート1：まず現状を確認してほしい

### 使う場面
- これから作業を始める前に、対象フォルダ・ファイルの状態を把握したい
- 人から引き継いだばかりで、何があるか分からない
- 「とりあえず状況を教えて」と聞きたい時

### テンプレート本文
```
【依頼の目的】
{何のために現状確認するか（例：月次作業を始める前の棚卸し）}

【確認してほしい対象】
{ファイル・フォルダのパス、または対象範囲}

【確認してほしい観点】
- {観点1（例：ファイル数と合計サイズ）}
- {観点2（例：最終更新日が古いもの）}
- {観点3（例：命名規則に合わないもの）}

【成果物の形式】
箇条書きのサマリー。長くても画面1ページ分に収めてください。

【重要】
変更・削除は一切しないでください。読み取り・確認のみです。
```

### 記入のコツ
- 「観点」を先に決めるのがポイント。漠然と「現状教えて」だと返答も漠然とする
- 「変更しないで」を明示することで、意図しない編集を防げる

### 良い使用例
```
【依頼の目的】月次レポート作業を始める前に、input/ の状態を確認したい
【確認してほしい対象】共有ドライブ/01_月次レポート/2026_04/input/
【確認してほしい観点】
- CSVファイルの数と最終更新日
- ファイル名が命名規則（営業データ_YYYYMMDD.csv）に合っているか
- 先月（2026_03）に比べて不足しているデータがないか
【成果物の形式】箇条書きのサマリー
【重要】読み取りのみ。編集しないでください
```

### 悪い使用例
```
フォルダ見といて
```
- 目的・対象・観点がすべて不明

## テンプレート2：作業計画を出してほしい

### 使う場面
- いきなり作業させず、段取りだけ先に確認したい
- 手戻りを防ぎたい、危険な操作を含む作業の前

### テンプレート本文
```
【依頼の目的】
{これから何を達成したいか}

【対象ファイル・フォルダ】
{具体的なパス}

【制約・前提】
- {やってはいけないこと}
- {守ってほしい条件}

【成果物の形式】
{最終的に何が出来上がればよいか}

【重要】
作業はまだ始めないでください。
まず以下を含む「作業計画」だけを提示してください：
1. 実行する手順（番号付き、1ステップ1動作）
2. 各ステップでどのファイルに触るか
3. 想定されるリスクと、その回避策
4. 所要時間の目安
私が計画を確認してOKを出してから、実作業に進んでください。
```

### 記入のコツ
- 「計画に含めてほしい項目」を明示することで、計画の粒度が揃う
- 「私がOKを出してから」を書くことで、勝手に作業を始めることを防げる

### 良い使用例
```
【依頼の目的】先月分の営業レポートPDFを作成する
【対象】共有ドライブ/01_月次レポート/2026_03/
【制約】input/ の元CSVは絶対に上書きしない
【成果物】A4・5ページ以内のPDF
【重要】作業前に計画だけ先に見せてください（手順・リスク・所要時間を含む）
```

### 悪い使用例
```
レポート作って。あと計画も。
```
- 「計画も」だと作業と並行で進んでしまう可能性

## テンプレート3：ファイルを作成してほしい

### 使う場面
- 新規ファイルを1つ作ってほしい（テンプレート適用、雛形生成など）

### テンプレート本文
```
【依頼の目的】
{なぜこのファイルを作るか}

【作成するファイル】
- パス：{どこに置くか}
- ファイル名：{具体的な名前}
- 形式：{拡張子・文字コード}

【内容】
{ファイルに書くべき内容の概要。見出し構成や必須項目を列挙}

【参考にするもの】
{既存のテンプレート、過去のファイルなどがあればパスを指定}

【制約】
- 既存ファイルは上書きしないでください。同名ファイルがあれば、末尾に日付を付けてください
- {その他の制約}

【確認】
作業前に、作成するファイルの構成（見出しレベル）だけ先に提示してください
```

### 記入のコツ
- 「参考にするもの」があると、ゼロから生成されず、社内の既存書式に沿った出力になる
- 「同名ファイルがあったらどうするか」を必ず指定（上書き事故の防止）

### 良い使用例
```
【目的】来月の月次レポートの雛形ファイルを先に作っておきたい
【作成するファイル】
- パス：共有ドライブ/01_月次レポート/2026_05/
- ファイル名：月次レポート_202605_draft.xlsx
- 形式：xlsx
【内容】昨月（2026_04）のレポートと同じ構成。数値は空欄
【参考】共有ドライブ/01_月次レポート/2026_04/output/月次レポート_202604.xlsx
【制約】既存ファイルがある場合は作成せず、その旨を報告
```

### 悪い使用例
```
ファイル作って
```
- パス・名前・内容・形式すべて不明

## テンプレート4：既存ファイルを修正してほしい

### 使う場面
- 既にあるファイルの一部を書き換えたい
- 誤字修正、データ更新、フォーマット調整

### テンプレート本文
```
【依頼の目的】
{なぜ修正するか}

【対象ファイル】
{パス}

【修正箇所】
- {どこを（行番号・見出し・セル範囲などで特定）}
- {何をどう変えるか（Before → After）}

【修正しないでほしい箇所】
- {触ってほしくない範囲}

【制約】
- 必ずバックアップを取ってから編集してください
- バックアップは同じフォルダに「{元ファイル名}.bak_YYYYMMDD」で保存

【確認】
編集前に、差分のプレビュー（Before/After 一覧）を提示してください
```

### 記入のコツ
- 「修正しないでほしい箇所」を書くことで、意図しない巻き添え修正を防げる
- Before/After を先に出させると、実行前に意図確認ができる

### 良い使用例
```
【目的】取引先A社の社名表記を新社名に更新する
【対象】共有ドライブ/03_取引先リスト/master.xlsx
【修正箇所】B列の「株式会社A商事」→「A商事株式会社」に全置換
【修正しないで】A列の取引先コード、C列以降の担当者情報
【制約】バックアップを取ってから実行。変更箇所のリストを作業後に提示
```

### 悪い使用例
```
A社のところ直しといて
```
- 対象ファイル・修正方法・禁止範囲が不明

## テンプレート5：差分を確認してほしい

### 使う場面
- 作業後に「何が変わったか」を一覧で見たい
- 2つのファイル・バージョンを比較したい

### テンプレート本文
```
【依頼の目的】
{なぜ差分を確認したいか}

【比較対象】
- A：{ファイルパスまたはバージョン}
- B：{ファイルパスまたはバージョン}

【確認したい観点】
- {観点1（例：数値の変化）}
- {観点2（例：見出し構成の変更）}
- {観点3（例：削除された項目）}

【成果物の形式】
Before / After を並べた表 or 箇条書き。変更のない箇所は省略して構いません。

【制約】
読み取りのみ。ファイルは変更しないでください。
```

### 記入のコツ
- 「何を比較するか」の観点を絞ると、重要な差分だけが浮かび上がる
- 大量の差分が出そうな場合は、「観点別にまとめて」と指示する

### 良い使用例
```
【目的】先月と今月の月次レポートで、構成に差がないか確認したい
【比較対象】
- A：共有ドライブ/01_月次レポート/2026_03/output/月次レポート_202603.xlsx
- B：共有ドライブ/01_月次レポート/2026_04/output/月次レポート_202604.xlsx
【観点】見出し構成の違い、列の追加・削除、数式の変更
【成果物】箇条書き
```

### 悪い使用例
```
違うとこ教えて
```
- 対象・観点すべて不明

## テンプレート6：エラーの原因を調べてほしい

### 使う場面
- 作業中にエラーが出て、原因が分からない
- 「なぜ動かないか」を切り分けたい

### テンプレート本文
```
【依頼の目的】
{何をしていた時にエラーが出たか}

【発生したエラー】
- エラー文言：{原文そのまま}
- 発生場所：{ファイル・行・操作手順}
- 発生日時：{YYYY-MM-DD HH:MM}

【直前にやった操作】
1. {手順1}
2. {手順2}
3. {エラーが出た操作}

【試したこと】
- {自分で試した対処（あれば）}

【依頼内容】
原因の仮説を3つ挙げ、それぞれの確認方法を教えてください。
**原因特定が終わるまで、修正は実施しないでください。**

【制約】
ファイルを変更しないでください。調査のみ。
```

### 記入のコツ
- エラー文言は絶対に省略せず、原文コピペ
- 「仮説を3つ」と指定することで、思い込みの単一原因決めつけを防げる
- 修正実行と原因調査を分けるのが安全

### 良い使用例
```
【目的】月次集計マクロを実行したらエラーで止まった。原因を調べたい
【エラー】「実行時エラー '1004': アプリケーション定義またはオブジェクト定義のエラー」
【発生場所】月次集計.xlsm の「集計」ボタン押下時
【発生日時】2026-04-24 10:15
【直前の操作】
1. input/sales_data.csv を開いた
2. 何も編集せず閉じた
3. 月次集計.xlsm を開き「集計」ボタンを押した
【試したこと】ファイルを再起動して再実行 → 同じエラー
【依頼】原因仮説3つと確認方法。修正はまだしないでください
```

### 悪い使用例
```
エラー出た。直して
```
- エラー文言・発生状況・再現手順がすべて不明

## テンプレート7：再発防止策を作ってほしい

### 使う場面
- エラーやミスの原因が判明した後、「次に同じことが起きないように」する
- 事後の振り返りをして、運用ルールに反映したい

### テンプレート本文
```
【依頼の目的】
{今回発生した問題の再発防止策を作りたい}

【発生した問題の概要】
- 何が起きたか：{問題の内容}
- 影響範囲：{どこまで影響したか}
- 発見経緯：{どうやって気づいたか}

【判明した原因】
{調査で分かった根本原因}

【依頼内容】
以下の観点で再発防止策を提案してください。
1. 作業前チェック（作業開始時に何を確認するか）
2. 作業中チェック（途中で何を確認するか）
3. 作業後チェック（完了時に何を確認するか）
4. CLAUDE.md または memory.md に追記すべき項目

【成果物の形式】
- 各観点ごとに、具体的なチェック項目を箇条書きで
- CLAUDE.md / memory.md への追記案は、そのままコピペできる形で

【重要】
ファイルへの実際の追記はまだしないでください。提案だけにしてください。
```

### 記入のコツ
- 「原因」を明確にしてから対策を依頼する。原因が曖昧だと対策も的外れになる
- 「コピペできる形で」と指定すると、そのまま運用ルールに反映できる

### 良い使用例
```
【目的】先週の元CSV上書き事故の再発防止策を作りたい
【問題】作業中に Ctrl+S で元CSVを上書きし、復旧に半日かかった
【原因】作業用コピーを作らず、直接元ファイルを編集していた
【依頼】作業前・中・後のチェック項目を提案。CLAUDE.md への追記案もください
【重要】追記はまだしないでください
```

### 悪い使用例
```
気をつけるルール作って
```
- 問題の内容・原因・求める形式がすべて不明

## テンプレート8：スキル化してほしい

### 使う場面
- 繰り返している作業を定型化して、今後使い回したい
- 個人の頭の中にあるノウハウを、SKILL.md として言語化したい

### テンプレート本文
```
【依頼の目的】
{どの作業を skill 化したいか}

【対象業務】
- 業務名：{一言で}
- 実施頻度：{週1 / 月1 など}
- 所要時間：{目安}

【現在の手順（自分が知っている範囲）】
1. {手順1}
2. {手順2}
3. {...}

【失敗しがちなポイント】
- {気をつけていること、過去のつまずき}

【依頼内容】
上記を元に SKILL.md の骨組みを作ってください。
含めるセクション：
- スキル名
- 使う場面
- 前提条件
- 手順（番号付き、1ステップ1動作）
- 成果物
- 注意事項（失敗例）

【制約】
既存の skill フォルダに同名ファイルがある場合は、上書きせずにファイル名末尾に「_draft」を付けてください
```

### 記入のコツ
- 「自分が知っている範囲」で手順を書き出す。完璧でなくてよい
- 「失敗しがちなポイント」を入れると、skill の実用性が大幅に上がる

### 良い使用例
```
【目的】毎月やっている営業レポート作成を skill 化したい
【業務】月次営業レポート作成、月1回、約4時間
【現在の手順】
1. input/ の最新CSVをコピー
2. 千円単位に丸める
3. 前月比を計算
4. テンプレートにはめる
5. PDF出力
【失敗しがち】元CSVを直接編集してしまう、前月ゼロ行でのゼロ除算
【依頼】SKILL.md の骨組みを作成。同名ファイルがあれば _draft 付与
```

### 悪い使用例
```
なんかスキルにして
```
- 対象業務・手順・形式すべて不明

## テンプレート9：CLAUDE.mdに追記してほしい

### 使う場面
- 新しい運用ルールを CLAUDE.md に反映したい
- 追記内容を自分で考えるのが難しい時、叩き台を作ってほしい

### テンプレート本文
```
【依頼の目的】
{なぜ CLAUDE.md を更新するか}

【追記したい内容の要点】
{箇条書きで、新ルール・決定事項を列挙}

【追記先セクション】
{CLAUDE.md のどのセクションに入れるか}
（どこに入れるか分からない場合は「適切なセクションを提案してください」と書く）

【依頼内容】
上記を元に、CLAUDE.md に追記する文面案を作成してください。

【重要】
文面の提案のみで、CLAUDE.md 本体への書き込みはまだしないでください。
提案を確認したうえで、私から改めて反映指示を出します。
更新履歴への追記文も一緒に用意してください（日付・変更概要）。
```

### 記入のコツ
- 「提案のみ。本体への反映は別途指示」を明示することで、意図しない上書きを防げる
- 「更新履歴」への追記も同時に依頼しておくと、メンテが続きやすい

### 良い使用例
```
【目的】先週の上書き事故を受けて、運用ルールを追加したい
【追記内容】
- 作業開始時は必ず元ファイルのコピーを作る
- コピーのファイル名は「{元ファイル名}_work_YYYYMMDD」とする
【追記先】「9. 絶対に削除・変更してはいけないファイル」セクションの下
【依頼】文面案と、更新履歴への追記文を用意。本体反映はまだしないでください
```

### 悪い使用例
```
CLAUDE.md 更新しといて
```
- 何を・どこに・どう追記するか不明

## テンプレート10：作業完了報告をしてほしい

### 使う場面
- 一連の作業が終わったので、何をしたか・結果どうなったかをまとめてほしい
- 上司・関係者への報告テキストを作ってもらいたい

### テンプレート本文
```
【依頼の目的】
{誰に報告するか、何のための報告か}

【報告対象の作業】
- 作業名：{一言で}
- 実施日時：{YYYY-MM-DD HH:MM 〜}
- 対象ファイル・フォルダ：{パス}

【報告に含めてほしい項目】
1. 実施した作業の概要（3行以内）
2. 生成・変更したファイル一覧
3. 想定外の事象があればその内容と対処
4. 残課題・次回やること
5. 所要時間

【報告先・形式】
- 報告先：{Slack / メール / 口頭 など}
- 形式：{箇条書き / 文章 / 表}
- 分量：{目安の文字数や行数}

【文体】
{ですます / 体言止め可 / 敬語レベル}

【重要】
個人名・社外秘情報の扱いに注意してください。
{マスキング対象があれば具体的に指示}
```

### 記入のコツ
- 報告先によって文体・分量が変わる（社内Slackと社外メールでは全く違う）
- 「残課題」「次回やること」を入れると、報告を受け取った側の次アクションが明確になる

### 良い使用例
```
【目的】上司の鈴木室長に、今月の月次レポート作業完了を報告
【作業】月次営業レポート2026年4月分の作成
【日時】2026-04-24 09:00 〜 13:00
【対象】共有ドライブ/01_月次レポート/2026_04/
【含めてほしい項目】実施概要、生成ファイル、想定外事象、残課題、所要時間
【報告先・形式】Slack DM、箇条書き、300字以内
【文体】ですます。結論ファースト
【重要】取引先名は伏せ字（A社、B社）に
```
