# Claude Code 基礎勉強会 受講者ハンドアウト

> このハンドアウトは、勉強会の受講後にもう一度見返して、自分の業務で使い始めるためのものです。
> **注意：このハンドアウトに書いてあることが仕様と異なる場合は、公式情報を優先してください。Claude Codeの仕様は日々更新されるため、確信が持てない部分は「要確認」と明記しています。**

---

## 1. はじめに（この資料の使い方）

この資料は、勉強会で話した内容を後からでも一人で確認できるようにまとめたものです。以下のような使い方を想定しています。

- 勉強会の翌日、最初の10分間でざっと読み直す
- 自分の業務でClaude Codeを使う前に、該当箇所だけ確認する
- 同僚に共有して、社内での普及ツールとして使う

A4サイズで印刷する前提でレイアウトしているため、紙に出しても読みやすいはずです。

> **大事なこと：**
> この資料に書かれている内容は、2026年時点の勉強会担当者の理解に基づくものです。
> 実際にツールを使う際は、必ず公式ドキュメントと社内の最新ルールを確認してください。

---

## 2. 今日のゴール（到達目標の再掲）

勉強会で到達したかった3つのゴールです。

1. Claude Codeに作業を頼むときの「基本の頼み方」を言葉で説明できる
2. CLAUDE.md、memory、skills、MCP、hooks、subagents の違いをざっくり区別できる
3. やってはいけないことを知り、事故を避けられる

すべて完璧である必要はありません。「聞いたことがある」「どこを見れば載っているか知っている」状態なら十分です。

## 3. 用語集

勉強会で登場したキーワードを、業務スタッフ目線でまとめました。正確な技術仕様については、該当の公式ドキュメントを確認してください。

### Claude Code
Anthropic社が提供する、PC上で動くAIアシスタント。チャット版のAIと違い、あなたのPCのファイルを直接読み書きできる点が最大の特徴。ターミナル（黒い画面）で日本語の指示を受け付ける。

### セッション
Claude Codeを起動してから終了するまでの「1回分の会話の箱」。セッション内では会話を覚えているが、セッションを閉じると基本的にリセットされる。打ち合わせ1回分、と捉えるとイメージしやすい。

### CLAUDE.md
プロジェクトのフォルダに置いておく、人間がClaude向けに書く**取扱説明書**。ここにルールや禁止事項、好みを書いておくと、Claude Codeが起動するたびに参照してくれる。たとえ話：**新人バイトに渡す店舗マニュアル**。

### memory / Auto memory
Claude側が「作業中に気づいたこと」を自分でメモしておく仕組み。CLAUDE.mdが人間が書くマニュアルなのに対して、memoryはClaude側の手帳。たとえ話：**バイトが自分の手帳につけている作業中の注意点**。保存場所や挙動の詳細はバージョン・環境で異なるため要確認。

### skills / SKILL.md
特定の作業に特化した**専用レシピ**。たとえば「議事録整形」「請求書読み取り」など、型の決まった作業をまとめて登録しておける。skillの中に `SKILL.md` というファイルを置き、frontmatter（ファイル冒頭の設定ブロック）で使いどころを書く。仕様の詳細は公式ドキュメントで要確認。

### frontmatter
マークダウンファイルの冒頭に置かれる「設定ブロック」。`---` で挟まれた区間に、名前・用途・適用条件などをキーと値のペアで書く。SKILL.mdなどでよく使われる。

### MCP
Model Context Protocolの略。Claude Codeと外部サービス（Googleドライブ・Slack・社内DBなど）を**つなぐための接続口の規格**。たとえ話：**家電のコンセント規格**。業務スタッフは「自分の環境がどこに繋がっているか」を把握しておけば十分。

### hooks
**特定のタイミングで自動的に動くスイッチ**。たとえば「ファイル保存時に自動でバックアップを取る」といった自動処理を設定できる。たとえ話：**センサーライト**。自分で設定するのは中級者以上の話なので、まずは「そういう仕組みがある」と知っておけばOK。

### subagents
メインのClaude Codeが**別のClaude Codeに仕事を振れる仕組み**。並行作業や役割分担ができる。たとえ話：**別担当のAIにタスクを渡す**。PMが複数の担当者にタスクを振るイメージ。

### プロンプト
Claudeへの**指示文そのもの**。あなたがターミナルに書き込む1回分のお願い。

### コンテキスト
Claudeが**今見えている情報全体**。プロンプトだけでなく、CLAUDE.mdの内容、開いているファイルの中身、これまでの会話履歴などが全部含まれる。上限があり、長く会話を続けると昔の話を忘れることがある（具体的な上限値は要確認）。

### エージェント
自分で考えて手を動かすAIのこと。Claude Code自体が代表例。

### リポジトリ / プロジェクト
**プロジェクトのフォルダ全体**。業務で言えば「1案件のフォルダ」くらいの感覚でOK。厳密にはGit管理されたフォルダを指すが、ここではゆるく同義として扱う。

### コマンド実行
Claude Codeが裏でPCのコマンドを実行すること。ファイル一覧取得・移動・削除などが含まれる。実行前に確認画面が出ることがあるので、内容を**読んでから**Yesを押すこと。

## 4. CLAUDE.md / memory / skills / MCP / hooks / subagents の違い

それぞれの役割を1枚の表で把握できるようにまとめました。
詳細な仕様は公式ドキュメントで確認してください。

| 種類 | 書く人 | 書く内容 | 向いている情報 | 向いていない情報 | 読み込まれるタイミング |
|---|---|---|---|---|---|
| CLAUDE.md | 人間 | プロジェクト全体のルール・禁止事項・言葉づかいの好み | 常に守ってほしいルール／触ってほしくないファイル／成果物の置き場所 | 一時的なメモ／長大な参考資料／秘密情報 | セッション開始時にプロジェクトのCLAUDE.mdが読み込まれる（詳細挙動は要確認） |
| memory / Auto memory | Claude（学習） | 作業中に気づいた環境情報・過去の傾向など | Claude側の気づき・再発防止メモ | 人間側で確実に守らせたいルール（それはCLAUDE.mdへ） | 自動的に参照される想定（挙動・保存場所はバージョンで異なるため要確認） |
| skills | 人間 | 特定作業の専用レシピ（SKILL.md＋frontmatter） | 繰り返し発生する定型業務／フォーマットが決まった処理 | 一度きりの作業／ルール全体 | 該当作業の指示があった時に呼び出される（詳細な発火条件は要確認） |
| MCP | 人間（設定） | 外部サービスへの接続設定 | Googleドライブ・Slack・社内DBなどとの連携 | Claude Code内の挙動ルール | MCPが設定されているサービスへアクセスが必要になったタイミング |
| hooks | 人間（設定） | 特定タイミングで自動実行する処理の設定 | 保存時の自動バックアップ／実行前の検査など | 「こう書いてほしい」という指示的な内容 | 設定したイベントが発生した時（詳細な発火条件は要確認） |
| subagents | 人間（設計） | 別のClaudeに任せるタスクの設計・役割分担 | 並行調査／重い下調べ／別視点でのレビュー | 小さな一発作業／対話が必要な作業 | メインのClaudeが必要と判断した時、もしくは人間が明示的に呼び出した時 |

> **覚え方のコツ：**
> - **CLAUDE.md** → 取扱説明書（人間が書くルール）
> - **memory** → 作業中の注意点（Claudeの手帳）
> - **skills** → 専用レシピ（特定業務の手順書）
> - **MCP** → 接続口（外部サービスのコンセント）
> - **hooks** → 自動スイッチ（センサーライト）
> - **subagents** → 別担当AI（タスクを振る相手）

## 5. 作業依頼テンプレート集（コピペして使える6種類）

そのままコピーして、カギカッコの中を置き換えるだけで使えます。

### テンプレート1：現状確認してほしい

```
次の作業の前に、現状を確認してください。

対象フォルダ：[フォルダのパス]
確認してほしいこと：
- フォルダ内のファイル一覧（種類ごとにまとめて）
- 気になる点・通常と違う点があれば指摘
- 次に何をすべきかの提案

この時点ではファイルは編集しないでください。確認結果だけを返してください。
```

### テンプレート2：作業計画を出してほしい

```
次の作業をお願いしたいです。いきなり作業に入る前に、作業計画を立ててから提示してください。

やりたいこと：[ゴール]
前提条件：[制約・ルール]
完了の定義：[どうなったら完了とみなすか]

計画には以下を含めてください：
- 作業ステップの一覧（順序付き）
- 各ステップで触るファイル
- 想定されるリスクと代替案

計画を見て私がOKを出したら、実作業に進んでください。
```

### テンプレート3：ファイル作成してほしい

```
新しいファイルを作成してください。

作成先：[保存先のパス]
ファイル名：[ファイル名]
形式：[Markdown／テキスト／CSV など]
内容：
- [項目1]
- [項目2]
- [項目3]

既存ファイルには触らないでください。
```

### テンプレート4：既存ファイル修正してほしい

```
既存ファイルを修正してください。

対象ファイル：[ファイルのパス]
修正内容：
- [変更点1]
- [変更点2]

注意事項：
- 修正前に現在の内容を確認・要約して私に見せてください
- 私のOKを受けてから書き換えてください
- それ以外の箇所は変更しないでください
```

### テンプレート5：差分確認してほしい

```
これまでの変更内容の差分を確認したいです。

対象：[ファイル名 または フォルダ]
確認したいこと：
- どのファイルがいつ変更されたか
- 変更前と変更後の内容の違い（要約でOK）
- 気になる変更があれば指摘

この時点では追加の変更をしないでください。差分の報告のみをお願いします。
```

### テンプレート6：完了報告してほしい

```
作業完了の報告をお願いします。

報告に含めてほしい項目：
- 実施した作業内容（箇条書き）
- 変更したファイルの一覧
- 未実施・未確認の項目があれば明記
- 次に私がやるべきこと

完了報告は、簡潔に・事実ベースでお願いします。
```

## 6. 初心者向けチェックリスト

### 作業前チェックリスト

- [ ] Claude Codeを起動するフォルダは正しいか
- [ ] 対象のファイル・フォルダのバックアップは取ったか（大きな作業の前は必須）
- [ ] 触ってほしくないファイル・フォルダをCLAUDE.mdに書いたか、または依頼文で明示したか
- [ ] 依頼文に「何を」「どこに」「どういう形で」の3つが含まれているか
- [ ] 機密情報・個人情報を渡さない設計になっているか

### 作業中チェックリスト

- [ ] 確認画面が出たら、必ず内容を読んでからYes／Noを判断しているか
- [ ] 削除系・移動系のコマンドは範囲指定が広すぎないか
- [ ] Claudeの応答が想定と違ったら、指示を書き直して再試行しているか
- [ ] 長い会話の途中で「コンテキストが埋まっているかも」と感じたら、一度整理して再開したか

### 作業後チェックリスト

- [ ] 成果物を人間の目で最終確認したか
- [ ] 意図しない変更が他のファイルに起きていないか
- [ ] うまくいかなかった指示があれば「どう直せばよかったか」メモしたか
- [ ] 今回使ったCLAUDE.md・依頼文のうち、次回も使えるものを保存したか

## 7. やってはいけないこと集

### 絶対に避けること

- **機密情報・個人情報をそのまま投げる**
  社内の情報取扱ルールと、Claude Codeの利用プランの規約を事前に必ず確認。判断がつかないなら使わないのが安全。

- **確認画面を無心でYesし続ける**
  削除・移動・上書きは不可逆なことが多い。確認画面はコマンドの種類・対象・範囲指定を読んで判断する。

- **バックアップなしで大きな作業をさせる**
  「このフォルダ全部整理して」のような依頼をいきなり実行させない。先にフォルダごとコピーを取ってから進める。

- **「いい感じにやっといて」と丸投げ**
  ゴールと制約を明示しないと、見当違いの成果物が出てくる。最低限「何を」「どこに」「どういう形で」を指定する。

### できれば避けたいこと

- 複数のClaude Codeセッションで同じファイルを同時に編集する（競合の原因）
- CLAUDE.mdに大量の情報を詰め込む（要点が薄まる）
- 自分が理解していないコマンドをYesで通し続ける
- 社内ルールが曖昧なまま使い続ける（使う前にルール整備を相談）

## 8. 困った時の対処法フローチャート（文章版）

困ったときは、次の順番で対処してみてください。

1. **そもそもセッションが生きているか確認する**
   Claude Codeが反応しない場合、まず一度セッションを終了し、再起動してもう一度依頼を試みる。これだけで解決することが多い。

2. **依頼文を読み直す**
   同じ依頼文をもう一度見て、「何を」「どこに」「どういう形で」が揃っているか確認。曖昧な表現があれば具体化する。

3. **小さい単位で試す**
   大きな依頼で失敗したら、そのタスクの一番小さい部分だけ切り出して頼んでみる。部分的に動けば、残りを追加で頼んでいける。

4. **Claudeに自分で状況を説明させる**
   「いま何が起きているか、今あなたが持っている情報をまとめて教えてください」と聞くと、認識のズレが見つかることがある。

5. **意図しない変更が起きたと感じたら、一度手を止める**
   追加依頼を繰り返すと傷が深くなる。まず差分を確認し、元ファイルのバックアップやPCのゴミ箱・バージョン管理から復元を検討する。

6. **社内の詳しい人に聞く**
   セッションの画面や依頼文をそのまま見せると話が早い。恥ずかしがらず共有する。

7. **公式ドキュメントを確認する**
   仕様はアップデートで変わるため、最終的には公式情報が頼り。このハンドアウトの内容と食い違う場合は公式優先。

> **フローチャート要約（覚え書き）：**
> 再起動 → 依頼文見直し → 小さく切る → Claudeに状況を言わせる → 手を止める → 人に聞く → 公式で確認

## 9. 次のステップ

勉強会で終わらせずに、自分の業務で使ってみるのが一番の定着方法です。以下の順でチャレンジしてみてください。

1. **自分の業務フォルダに小さなCLAUDE.mdを1つ置く**
   10行程度でOK。「このフォルダの目的」「触ってほしくないファイル」「成果物の置き場所」だけで十分。使いながら直していけばいい。

2. **1週間、軽作業を1日1つ頼んでみる**
   大きな作業でなくてOK。「このフォルダのファイル一覧を整理して」「このメモから要点を3つ抜き出して」など、成功しやすいサイズで。

3. **うまくいかなかった指示を記録する**
   どう直せばよかったかを書き留めておくと、自分専用の「依頼文テンプレ集」が育っていく。

4. **社内で成功事例・失敗事例をシェアする**
   「こう頼んだらうまくいった」「こんな失敗をした」を共有することで、社内全体の練度が上がる。次の勉強会のネタにもなる。

5. **慣れてきたら、自分の定型業務をskillsにまとめる**
   これは中級編。社内で詳しい人に相談しながら進めるのがおすすめ。

## 10. 参考：公式情報への導線

このハンドアウトは入門者向けに分かりやすさを優先しているため、厳密な仕様は省略したり簡略化した部分があります。
**正確な仕様・最新の機能については、必ず以下を確認してください。**

- Anthropic社 公式サイト（Claude Code関連の公式ドキュメント）
- 社内のIT担当／管理部門が整備した社内ガイドライン
- Claude Code自身に `/help` 等のコマンドで最新情報を尋ねる（コマンドの存在・挙動は要確認）

> **重要：**
> Claude Codeは機能追加・仕様変更のペースが速いツールです。
> 「昔こう書いてあった」「昔はこう動いた」が通用しないことも普通にあります。
> **違和感があったら公式情報で確認する**習慣をつけてください。
> そして、このハンドアウトの内容と食い違う場合は、**公式情報を優先**してください。
