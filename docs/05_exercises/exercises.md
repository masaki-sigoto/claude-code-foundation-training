# Claude Code基礎勉強会 演習集

本演習集は、非エンジニアの業務スタッフが Claude Code を業務で使いこなせるようになることを目的としています。
全5演習を通じて、「指示文クッション方式」を実践しながら、業務寄りの作業（議事録整形・マニュアル作成・フォルダ整理・CSV作成・簡易レポート）を体験します。

## 全演習で使う基本概念

### 統一比喩
- **Claude Code** = 机に来て作業してくれるアシスタント。指示があれば動き、なければ待機する。
- **CLAUDE.md** = 家事代行さんに渡す「家のルール」。共通の前提を書いておく紙。
- **memory** = 相手の好みメモ。個別のクセや繰り返し依頼を記録する付箋。
- **skills** = 料理のレシピ。手順が決まっている作業を「型」として渡す。

> Claude Code の仕様や機能名・保存場所は更新されることがあります。本文中の挙動説明は2026年4月時点のもので、必ず最新の公式情報で確認してください。

### 指示文クッション方式とは

雑な要望を、いきなり Claude Code に渡すのではなく、間に「通常チャットAI（ChatGPT・Gemini・Claude のチャット版など）」を挟んで指示文を整える方法です。

```
[人間の頭の中の雑な要望]
       ↓
[ChatGPT / Gemini / Claude（チャット版）で整形]
       ↓
[整った指示文を Claude Code に渡す]
       ↓
[Claude Code から途中報告]
       ↓
[途中報告をチャットAIに貼って壁打ち]
       ↓
[次の指示を作って Claude Code に戻す]
```

ポイントは「Claude Code は実行が得意・チャットAIは整理が得意」という役割分担です。
本演習集では、全5演習でこの方式を実装例として示します。

### 用語の補足
- **Markdown（マークダウン）**：見出しや箇条書きを記号（`#`や`-`）で表す軽量な書式。`.md` 拡張子のテキスト。
- **CSV（シーエスブイ）**：`,`で区切られた表形式テキスト。Excel等で開ける。
- **テンプレート**：何度も使い回せる雛形。

## 演習1：議事録からプレゼン資料の構成を作る

- 所要時間：15分
- 難易度：初級
- 学ぶこと：
  - 作業前計画（いきなり本文を書かせず、まず構成を作らせる）
  - 生成物の段階分け（構成 → 本文 → 仕上げ）
  - 指示文クッション方式の基本サイクル
  - 完了報告の確認方法

### 目的
短い議事録メモから、社内勉強会で使うプレゼン資料の「構成（目次）」だけをまず作らせ、確認のうえで本文作成へ進む流れを体験する。

### 背景
受講者が最も失敗しやすいのは「いきなり完成形を作って」と頼んでしまうことです。
プレゼン資料のような長い成果物は、構成 → 本文 → 仕上げ、の段階に分けて指示するほうが、修正コストが低くなります。

### 使用するサンプルデータ
以下の5分間ミーティングメモを使います。

```
【日時】2026年4月15日(水) 14:00〜14:20
【場所】会議室B
【参加者】山田、佐藤、鈴木
【議題】次回の社内勉強会の内容について

- テーマは「業務でのAI活用」に決定
- 対象：非エンジニアの全社員(約30名)
- 所要時間：60分
- 構成案：
  - AIで何ができるか(15分)
  - 実際に試した事例3つ(20分)
  - 質疑応答(15分)
  - まとめ(10分)
- 注意点：
  - 専門用語は避ける
  - 失敗例も入れたい
  - スライドは20枚以内に
- 次回MTG：4/22 同じ時間帯
```

### 受講者がやること（手順）
1. 上記メモをテキストファイルに保存する
2. Claude Code に「構成だけ」を作るよう依頼する
3. 出てきた構成を確認し、必要なら修正依頼を出す
4. 構成が固まったら、本文作成を依頼する
5. 完了報告を確認し、抜け漏れがないか点検する

### 悪い指示の例
受講者が最初にやりがちな雑な指示です。

```
このメモからプレゼン資料を作って
```

問題点：
- 何枚作るのか不明
- 構成と本文どちらを期待しているか不明
- 出力形式（Markdown? PowerPoint?）不明
- 確認タイミングが指定されていない

### 良い指示の例（直接書く場合）

```
添付メモを元に、社内勉強会用のプレゼン資料の「構成案」だけ作ってください。

- 目的：4月22日の社内勉強会で配布するスライド20枚分の構成
- 対象読者：非エンジニアの全社員30名
- 制約：
  - スライド枚数は20枚以内
  - 1スライド1メッセージ
  - 専門用語は使わない
- 成果物：Markdown形式で、各スライドのタイトルと「3行以内の要旨」のみ
- 確認タイミング：構成案ができた時点で一旦止めて私に見せてください。本文はまだ書かないでください。
```

### 指示文クッション方式での改善例

**ステップ1：チャットAIへ依頼する文（口頭メモを整形してもらう）**

```
【依頼】次の私の口頭メモを、Claude Code に渡せる丁寧な指示文に整形してください。

口頭メモ：
- 議事録から勉強会のプレゼン作りたい
- でもいきなり本文じゃなくて構成だけ
- 20枚以内
- 非エンジニア向け
- 確認してから本文に進みたい

整形時の注意：
- 目的・対象・制約・成果物・確認タイミングを明記
- 一度に全部やらせない（構成→本文の二段階）
- 出力形式はMarkdown
```

**ステップ2：チャットAIから返ってくる整った指示文（例）**

```
以下のメモから、社内勉強会用プレゼンの「構成案のみ」を作成してください。

【目的】2026/4/22の勉強会(60分・対象30名)で使うスライド20枚以内の構成案
【対象読者】非エンジニアの全社員
【制約】
- スライドは20枚以内
- 1スライド1メッセージ
- 専門用語は避ける
- 失敗例も含める
【成果物】
- Markdown形式
- スライド番号 / タイトル / 3行以内の要旨
【確認タイミング】
- 構成案を出した時点で必ず一度止めて、私の確認を待ってください
- 本文(各スライドの詳細)はこの段階では書かないでください

【入力メモ】
(ここに議事録メモを貼り付け)
```

**ステップ3：Claude Codeに貼る**
ステップ2の整った指示文を Claude Code にそのまま貼り付けて実行します。

**分岐点での使い方（途中報告 → 壁打ち）**

Claude Code から次のような途中報告が返ってきたとします。

```
構成案を作成しました。20枚で組んでいます。
1. タイトル
2. 自己紹介
3. AIで何ができるか(全体像)
...(中略)
20. まとめと次のアクション
本文に進めてよろしいでしょうか?
```

ここで一度チャットAIに相談します。

```
【相談】Claude Code から以下の構成案が返ってきました。
元のメモにあった「失敗例」が反映されていない気がします。
どこに何枚入れるのが自然か、修正案をください。
あわせて Claude Code に渡す追加指示文も作ってください。

(ここに構成案を貼る)
```

チャットAIから修正案と追加指示文を受け取り、それを Claude Code に渡して構成を更新してもらいます。

### 提出物・成果物
- 構成案（Markdownファイル1つ）
- 確認後に本文を含めた完全版（Markdownファイル1つ）
- チャットAIとのやり取りログ（簡易メモで可）

### 自己チェックリスト
- [ ] 構成案の段階で一度止まったか
- [ ] スライド枚数が20枚以内か
- [ ] 失敗例が含まれているか
- [ ] 専門用語が使われていないか
- [ ] 完了報告で「やったこと」を確認したか

### 良い回答例（抜粋イメージ）

```
# 社内勉強会「業務でのAI活用」構成案 (全18枚)

## 1. タイトル
- タイトル / 日付 / 対象

## 2. 今日のゴール
- 「明日から1つ試せる状態」を目指す

## 3. AIで何ができるか(全体像)
- 文章作成 / 整理 / 集計の3カテゴリ
...
```

### 悪い回答例（よくある失敗）
- いきなり全スライドの本文まで書かれてしまう
- スライド枚数が増えすぎている（目安：30枚を大きく超える）
- 「AI活用とは何か」など抽象論で枚数を稼いでいる
- 失敗例が一切入っていない

### 講師向けガイド

**進行の目安**
- 0〜2分：演習説明とサンプルメモ配布
- 2〜5分：チャットAIで指示文を整える
- 5〜10分：Claude Code で構成案作成 → 確認 → 修正
- 10〜13分：本文作成依頼
- 13〜15分：共有タイム

**巡回で見るポイント**
- 構成段階で止めずに本文まで進ませていないか
- チャットAI側の指示文に「確認タイミング」が入っているか
- 受講者がClaude Codeの完了報告を読み飛ばしていないか

**失敗例の拾い方**
- 「構成と本文を一気に出させた人いますか？」と挙手で聞く
- 雑な指示で出した結果と、整えた指示で出した結果を比較投影

**共有タイムでの引き出し方**
- 「クッションを挟んで何が変わりましたか？」
- 「Claude Code の途中報告を信用しましたか？それとも確認しましたか？」

### 発展課題
- 同じメモから「経営層向け5枚版」も作ってみる
- 構成段階で章立てを3パターン提案させ、選ぶ練習をする

## 演習2：簡単な業務マニュアルを作る

- 所要時間：12分
- 難易度：初級
- 学ぶこと：
  - 情報整理（箇条書きから読み物への変換）
  - 読み手設定（初心者か経験者か）
  - 文章のトーン指定
  - テンプレート化（次回以降も使える形）

### 目的
箇条書きの業務手順を、初心者でも迷わず実行できるマニュアルに整形する。
注意点・チェックリスト・よくあるミスもセットで揃える。

### 背景
業務マニュアルは「自分が分かっている前提で書いてしまう」のが最大の問題です。
読み手を明示し、トーンを揃え、テンプレートにすることで、属人化を防げます。

### 使用するサンプルデータ
以下の備品発注手順メモを使います。

```
【備品発注手順メモ】
- 在庫が少なくなった備品を発見したら総務に連絡
- 発注は基本Amazonビジネス
- 5000円超は上長承認が必要
- 承認はSlackの#approvalチャンネル
- 発注後、注文番号を備品台帳(Googleスプレッドシート)に記入
- 届いたら検品して問題なければ伝票にハンコ
- 月末に発注一覧を経理に提出

注意:
- トナー類は型番ミス多いので必ず2回確認
- 急ぎの場合でも上長承認は省略しない
- 個人カードでの立替は原則禁止
```

### 受講者がやること（手順）
1. 上記メモをテキストファイルに保存する
2. 「対象読者：入社1ヶ月以内の新人」と指定してマニュアル化を依頼する
3. 注意点とチェックリストを別セクションで出させる
4. 「よくあるミスTop3」を追記させる
5. 次回も使えるテンプレート部分を分離させる

### 悪い指示の例

```
このメモをマニュアルにして
```

問題点：
- 読み手が誰か不明
- トーン（です・ます/だ・である）の指定なし
- どこまで詳しく書くか不明

### 良い指示の例（直接書く場合）

```
添付の備品発注手順メモを、初心者向けマニュアルに整形してください。

- 対象読者：入社1ヶ月以内の新人(社会人経験あり、業務システムは初見)
- トーン：です・ます調、やわらかく
- 構成：
  1. このマニュアルの目的
  2. 事前準備(必要なアカウント等)
  3. 手順(番号付き、1ステップ1動作)
  4. 注意点
  5. チェックリスト(発注完了時に確認する項目)
  6. よくあるミスTop3と対処法
- 出力形式：Markdown
- 「テンプレートとして再利用できる空欄部分」も別ブロックで提示してください
```

### 指示文クッション方式での改善例

**ステップ1：チャットAIへ依頼する文**

```
【依頼】次の口頭メモを、Claude Code 用のマニュアル作成指示文に整えてください。

口頭メモ：
- 備品発注の手順をマニュアル化したい
- 新人向け
- やわらかい口調
- チェックリストも欲しい
- ミスしやすい点も書いてほしい
- 次の人が使い回せる雛形も別で

注文：
- 対象読者・トーン・構成・出力形式を必ず指定
- 「こうしてはいけない」も含める
```

**ステップ2：チャットAIから返ってくる整った指示文（例）**

```
以下の備品発注手順メモを、新人向けマニュアルとして整形してください。

【対象読者】入社1ヶ月以内の新人(業務システム初見)
【トーン】です・ます調、命令口調を避ける
【構成】
1. このマニュアルの目的
2. 事前準備
3. 手順(1ステップ1動作、番号付き)
4. 注意点
5. 完了時チェックリスト
6. よくあるミスTop3と対処法
7. テンプレート(空欄付きで再利用可能な雛形)

【出力形式】Markdown
【禁止】
- 専門用語の無補足使用
- 手順の中に注意書きを混ぜない(注意点セクションに分離)

【入力メモ】
(ここに備品発注メモを貼り付け)
```

**ステップ3：Claude Codeに貼る**
そのまま実行します。

**分岐点での使い方**

Claude Code から「マニュアル完成しました」と返ってきたら、チャットAIに以下のように相談します。

```
【相談】Claude Code が以下のマニュアルを出してきました。
新人が初見で読んでつまずきそうな箇所を3つ挙げて、修正指示文を作ってください。

(マニュアル本文を貼る)
```

返ってきた修正指示を Claude Code に渡して改善版を作成。

### 提出物・成果物
- マニュアル本体（Markdown）
- テンプレート部分（再利用可能な雛形）

### 自己チェックリスト
- [ ] 対象読者が冒頭に書かれているか
- [ ] 1ステップ1動作になっているか
- [ ] チェックリストが独立しているか
- [ ] よくあるミスTop3が具体的か
- [ ] テンプレート部分が空欄として明示されているか

### 良い回答例（抜粋イメージ）

```
# 備品発注マニュアル(新人向け)

## このマニュアルの目的
備品が少なくなったときに、新人の方が一人で発注完了まで進められるようにします。

## 事前準備
- Amazonビジネスのアカウント発行(総務に依頼)
- Slack #approval チャンネルへの参加
- 備品台帳(Googleスプレッドシート)の閲覧権限

## 手順
1. 在庫が少ない備品を見つけたら、総務に口頭またはSlackで連絡します
2. 総務から「発注OK」の返事をもらいます
...

## よくあるミスTop3
1. トナーの型番違い → 必ず2回確認しましょう
2. 5000円超なのに上長承認を省略 → 金額を見たら必ず承認フロー
3. 個人カードで立替 → 原則禁止です。困ったら総務へ
```

### 悪い回答例（よくある失敗）
- 「在庫が少なくなったら発注する」など、誰でも知っている粒度で終わる
- 注意点が手順の中に混ざって読みづらい
- チェックリストがなく、完了の判断ができない
- テンプレート部分が「ここに記入」とだけで雛形になっていない

### 講師向けガイド

**進行の目安**
- 0〜2分：説明とサンプル配布
- 2〜4分：チャットAIで指示文整形
- 4〜9分：Claude Codeで作成 → 修正
- 9〜12分：共有

**巡回で見るポイント**
- 「対象読者」の指定を忘れていないか
- 注意点が手順内に混ざっていないか
- テンプレートが本当に空欄として再利用できる形か

**失敗例の拾い方**
- 「読み手指定なしで出した人、結果どうでしたか？」
- 抽象的なマニュアルになった例を投影

**共有タイムでの引き出し方**
- 「自分の業務でテンプレ化したい手順は何ですか？」
- 「よくあるミスTop3は誰の知見が源泉ですか？」

### 発展課題
- 同じ内容で「ベテラン向け短縮版」も作る
- マニュアルを見て、CLAUDE.md に追記すべき社内ルールを抜き出す

## 演習3：フォルダ構成案を作る

- 所要時間：10分
- 難易度：初級
- 学ぶこと：
  - 安全な指示（提案だけで実行させない）
  - 触ってよい範囲の指定
  - 作業前確認（実行前に必ず一度止める）

### 目的
散らかったファイル一覧から、整理案だけをまず3パターン提案させ、選んだ案で構成図を作成する。
**実際のファイル移動は演習に含めない**。提案までで止める安全な指示の練習。

### 背景
ファイル整理を AI に頼むときの最大のリスクは「勝手に動かして元に戻せなくなる」ことです。
提案 → 確認 → 実行（人間の手で）の流れを体得します。

### 受講者がやること（手順）

## 演習4：CSVを作らせる

### 目的
手書き風の箇条書きメモから、カラム名と保存先を指定してCSV（カンマ区切りテキスト）を生成する。

### 背景

## 演習5：簡単な集計・予測レポートを作る

- 所要時間：12分
- 難易度：中級
- 学ぶこと：
  - データから読み取れることのコメント化
  - 断定しない表現（「〜の傾向が見える」「〜と推測される」）
  - レポートの段階構成（事実 → 解釈 → 仮説）

### 目的
過去6ヶ月の売上数字から、集計表 → 傾向コメント → 未来予測（仮説）の3段階レポートを作る。
**未来は断定しない**ことを徹底する。

### 背景
AIに数字を見せると、しばしば「断定的に未来を予測」してしまいます。
実務では「これは事実」「これは解釈」「これは仮説」を分けるのがレポートの基本作法です。

### 使用するサンプルデータ
以下の架空の売上データを使います。

```
【架空の月次売上(単位:万円)】

2025年11月: 820
2025年12月: 1050
2026年 1月:  610
2026年 2月:  720
2026年 3月:  890
2026年 4月:  940 (4/20時点 暫定)

参考メモ:
- 12月は毎年キャンペーンで増える傾向
- 1月は休暇影響で落ちる
- 4月は新年度で問い合わせ増
- 4月の数字は月途中の暫定値
```

### 受講者がやること（手順）
1. 上記データをテキストファイルに保存する
2. 集計表 → 傾向コメント → 未来予測（仮説）の3段構成でレポート作成を依頼する
3. 「未来は断定しない」「暫定値は明示」を制約にする
4. 完成後、断定的表現が混ざっていないかチェックさせる

### 悪い指示の例

```
この売上から来月どうなるか予測して
```

問題点：
- 事実と解釈と予測が混ざる
- 暫定値の扱いが指定されていない
- 断定OKと取られかねない

### 良い指示の例（直接書く場合）

```
添付の月次売上データから、3段構成のレポートを作成してください。

【構成】
1. 事実(集計表)：月別の数字を表にする。暫定値には(暫定)と明示
2. 解釈(傾向コメント)：3〜5項目、過去データで読み取れる傾向のみ
3. 仮説(来月の見立て)：あくまで仮説として、根拠と前提を添える

【表現ルール】
- 未来について断定しない(「〜になる」ではなく「〜になる可能性がある/〜と推測される」)
- 数字の出典・暫定/確定は必ず明示
- 「来月絶対こうなる」のような断言は禁止

【出力】Markdown
【最後に】レポート内の「断定表現が残っていないか」を自己チェックして報告
```

### 指示文クッション方式での改善例

**ステップ1：チャットAIへ依頼する文**

```
【依頼】Claude Code 用のレポート作成指示文を整えてください。

口頭メモ：
- 過去半年の売上データから集計+コメント+予測
- でも未来は断定NG(仮説として)
- 4月は暫定値なので明示
- 事実/解釈/仮説を分ける

注文：
- 表現ルールを強めに(断定禁止)

```

```

【3段構成】
1. 事実 - 集計表(月/売上/備考)
2. 解釈 - 過去データから読み取れる傾向(箇条書き3〜5項目)
```
