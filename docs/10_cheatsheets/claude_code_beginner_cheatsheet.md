# Claude Code 初心者チートシート

印刷してデスクに置く用・A4で1〜2ページ想定の凝縮版。
困ったらまずこれを見返してください。

---

## ■ 1. 最初にやること（3ステップ）

1. **プロジェクトフォルダを決める**（作業対象を1箇所に集約）
2. **CLAUDE.md を置く**（プロジェクトのルートに配置）
3. **作業依頼は「計画 → 確認 → 実行」の順で**（いきなり実行させない）

---

## ■ 1-2. 指示文クッション方式（迷ったらコレ）

▼ 雑な要望をそのままClaude Codeに投げない。**通常チャットAIをクッション（緩衝材）として挟む**のが鉄則。

```
■ 雑な要望はそのままClaude Codeに投げない
■ ChatGPT/Claude.ai/Geminiで「Claude Code向けの指示文に整えて」と頼む
■ 整った指示をClaude Codeに貼る
■ Claude Codeから途中報告が来たら、その報告をそのままチャットAIに貼って壁打ち
■ 「次にどう指示すべきか」を相談 → 整った次の指示をClaude Codeに貼る
```

● たとえ：引っ越し業者に「いい感じに運んで」ではなく**荷物リストと搬入先**を渡す／美容師に**写真と希望**を見せる
● 効果：人間の雑指示が整理される／Claude Codeが迷わない／作業範囲が明確／**分岐点での誤判断が減る**／結果的に早く正確
● 名称は固定：**「指示文クッション方式」**

---

## ■ 2. 作業依頼の基本形（コピペ用）

```
【目的】何のためにやるか
【対象】どのファイル・どのフォルダか
【制約】禁止事項・触らないもの・守ってほしいルール
【成果物】最終的にどんな状態になっていればOKか
【確認タイミング】作業前に計画を先に見せてください
```

● 迷ったらこのテンプレに埋めるだけ。曖昧依頼の事故を減らせます。

---

## ■ 3. CLAUDE.md に書くこと（10項目）

1. プロジェクトの**目的**
2. **重要フォルダ**とその役割
3. **命名規則**（ファイル名の付け方ルールなど。開発プロジェクトでは「ブランチ名」〔Gitでの作業分岐に付ける名前〕も含む）
4. **禁止事項**（触ってはいけないファイル・操作）
5. **報告形式**（箇条書き／表形式 など）
6. **文体ルール**（です・ます／敬語／絵文字の可否）
7. **削除禁止ファイル**の明示
8. **保存先**ルール（成果物はどこに置くか）
9. **確認手順**（大きな変更前に必ず計画を出す 等）
10. **注意事項**（過去の事故・再発防止メモ）

---

## ■ 4. skills に分けるもの（4つの目安）

▼ 以下に当てはまる作業は skills 候補

- **繰り返す作業**（週次・月次レポート等）
- **手順が長い作業**（工程が多く、CLAUDE.md だと読みにくい）
- **特定業務だけに関係する手順**（全体には不要）
- **CLAUDE.md が肥大化する原因**になっている内容

● CLAUDE.md＝総則、skills＝繰り返し作業の専用レシピ、のイメージ。

---

## ■ 5. やってはいけないこと（10個）

1. 指示が曖昧なまま実行させる
2. 確認せずに大量ファイル編集
3. 保存先を指定しない
4. 元ファイルのバックアップを取らず上書き
5. 完了報告を鵜呑みにする
6. MCP（外部ツールとの接続口）を権限確認なしで接続
7. memory に何でも（機密含めて）覚えさせる
8. **雑な指示を直接Claude Codeに投げる**（必ず指示文クッション方式でチャットAI経由で整える）
9. **分岐点で自己流で判断して進める**（途中報告はチャットAIに壁打ちしてから次の指示を出す）
10. **完了報告を確認せずに次へ進む**（ファイルパス・差分の要点を必ず目視確認）

---

## ■ 6. 便利な依頼文（7パターン）

● 「まず現状を確認して報告してください」
● 「実行前に計画だけ見せてください」
● 「変更する前に差分を示してください」
● 「この作業は skill 化できますか」
● 「この失敗の再発防止策を CLAUDE.md に追記してください」
● 「何を変更したか一覧で報告してください」
● 「不明点があれば先に質問してください」

---

## ■ 7. 完了報告で確認すること（5項目）

1. 何のファイルを変更したか（ファイルパス一覧）
2. 何を追加・削除したか（差分の要点）
3. 想定と違う点はないか
4. 次にやるべきことは何か
5. 残っている課題はあるか

● 「完了しました」だけの報告は信用しない。必ず内訳を確認。

---

## ■ 8. たとえ話サマリー（1行ずつ／揺らさない比喩）

- **Claude Code** ＝ 机に来て作業してくれるアシスタント（作業してくれるAIエージェント）
- **CLAUDE.md** ＝ 家事代行さんに渡す「家のルール」（プロジェクトの取扱説明書）
- **memory** ＝ 何度か来るうちに覚えた相手の好み（作業しながら覚えた注意点）
- **skills** ＝ 料理のレシピ（繰り返し作業の専用レシピ）
- **MCP** ＝ 外の部屋・倉庫への鍵（外部ツールとの接続口）
- **hooks** ＝ 玄関の鍵・財布チェック（決まったタイミングで動く自動スイッチ）
- **subagents** ＝ 引っ越しの荷造り担当・掃除担当（専門作業を任せる別担当AI）

● 比喩は教材全体で固定。違う言い方を聞いても、**この対応関係に戻して理解**してください。

---

## ■ 9. 困った時の1行フロー

▼ わからない → **公式ドキュメント確認**（一次情報が最優先。仕様の断定は避け、必ず公式情報で確認）
▼ 壊した → **いったん手を止めて状況確認**（慌てて追加操作しない）
▼ 出力が変 → **CLAUDE.md か指示を見直す**（前提がズレている可能性大）
▼ 雑な要望しか出てこない → **チャットAIで指示文を整えてから**Claude Codeに貼る（指示文クッション方式）
▼ Claude Codeが迷っている報告を出した → **その報告をチャットAIに貼って壁打ち**し、次の指示文を作る
▼ 大量変更指示が出そう → **いったん止める。「計画だけ先に出して」**と頼んで内容確認
▼ 削除を提案された → **必ず確認、勝手に消させない**。バックアップの有無を確かめる

---

## ■ 一言まとめ

**「計画→確認→実行→検証」** のサイクルを崩さない。
**雑な指示は指示文クッション方式（チャットAIで整形）で整える。**
**分岐点では途中報告をチャットAIに貼って壁打ちする。**
**迷ったら止めて人に相談。** これだけで事故の大半は防げます。
