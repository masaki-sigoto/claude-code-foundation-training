# 議事録タスク化スキル

## このスキルは何をするか

会議の議事録テキストを入力として、会議の中で発生した宿題・ネクストアクション・検討事項を抽出し、「担当者／内容／期限／報告先」が明確な形で一覧化します。発言の文脈を踏まえ、あいまいな宿題（「そのうち検討」など）は要確認として分離します。

## いつ使うか

- 会議が終わり、議事録テキストができあがった直後
- Slack などに散らかった会議メモから、正式なタスク一覧を整えたいとき
- 長時間の定例会（60 分以上）で、複数のネクストアクションが発生したとき
- 議事録担当が別の人で、自分が後からタスクだけ整理するとき

## 使ってはいけない場面

- 録音データ（音声ファイル）を直接文字起こしする場面（本スキルはテキスト前提）
- 人事評価・懲戒など機微な会議内容（一般的な業務タスクに限定）
- 会議途中のリアルタイム書き起こし（議事録が確定してから使う）

## 事前に準備するもの

- 議事録テキスト（Markdown／プレーンテキスト／Word のどれでも可）
- 会議の基本情報：日時、参加者、主な議題
- 出力先フォーマットの希望（Markdown 表／Notion 表／Slack 投稿用など）
- デフォルトの報告先（特に指定がない場合の既定担当者など）

## 手順

1. **議事録全体を 1 度通読する**
   いきなりタスクを拾わず、会議の流れと決定事項の位置関係を把握する。
2. **発言から「アクション性のある文」を抽出する**
   「〜する」「〜を確認する」「〜までにまとめる」などの表現を拾う。
3. **各アクションに 4 要素を当てはめる**
   - 誰が（担当者）
   - 何を（具体アクション）
   - いつまでに（期限）
   - どこで報告するか（報告先／報告方法）
4. **不足している要素を「要確認」として残す**
   例：担当者が「次回までに誰かが」で不明なら、担当者を「要確認」とする。勝手に推定しない。
5. **重複・矛盾を整理する**
   同じ内容が 2 箇所で言及されている場合は統合。期限が食い違う場合は要確認にする。
6. **優先度を 3 段階で付ける（任意）**
   高：期限が 1 週間以内／中：1 ヶ月以内／低：それ以降
7. **決定事項（タスクではない）を別欄にまとめる**
   「◯◯で合意」「△△の方針で進める」など、行動ではなく合意内容。
8. **要約の冒頭を添える**
   「本会議で発生したタスク計 X 件、うち要確認 Y 件」など。
9. **ユーザーに提示し、要確認項目の埋め方を相談する**

## 成果物の形式

```
# 会議タスク一覧

## 会議情報
- 日時: 
- 参加者: 
- 主要議題: 

## サマリー
- タスク計: X 件（うち要確認 Y 件）
- 決定事項: Z 件

## タスク一覧
| # | 担当者 | 内容 | 期限 | 報告先 | 優先度 |
| - | ------ | ---- | ---- | ------ | ------ |
| 1 | 山田 | 提案書 v2 作成 | 4/30 | 次回定例 | 高 |
| 2 | 要確認 | API 仕様書の最新化 | 5/15 | Slack #dev | 中 |

## 決定事項
- ◯◯の方針で進めることで合意
- △△は次四半期に先送り

## 要確認項目
- タスク #2: 担当者が議事録上で不明（「誰かが対応」との発言のみ）
```

## 品質チェック観点

- [ ] 議事録内のアクション発言が漏れなく拾われている
- [ ] 各タスクに担当者・内容・期限・報告先が揃っている（不足は「要確認」）
- [ ] 勝手に担当者を決めつけていない
- [ ] 決定事項とタスクが混在していない
- [ ] 同じ内容のタスクが重複していない
- [ ] 期限が日付で書かれている（「来週中」など曖昧な表現は日付に変換するか要確認に）
- [ ] 担当者名の表記が議事録と一致している

## よくある失敗

- **勝手に担当者を決める**：「多分これは営業部だろう」で書くと実害が出る。→ 不明なら必ず「要確認」。
- **決定事項とタスクを混ぜる**：「◯◯で合意」はタスクではない。→ 欄を分ける。
- **期限を「ASAP」のまま残す**：→ 具体日付にするか要確認に。
- **議事録にない内容を補完する**：良かれと思って「こうするべき」を足すのは NG。抽出に徹する。
- **細かすぎるタスク分解**：「資料を開く」レベルまで分解しない。成果物単位で 1 タスク。

## 使用例

### 入力例

> 昨日の開発定例の議事録です。タスク化お願いします。参加者は山田（PM）、佐藤（エンジニア）、鈴木（デザイナー）、中野（営業）。報告先は指定なければ Slack #dev にします。
>
> ---
> 4/23 開発定例
> - 山田：新機能 A の仕様書を来週月曜までに v2 に更新する
> - 佐藤：API のレスポンス遅延を調査、GW 明けまでに原因報告
> - 鈴木：ログイン画面のデザイン案を検討（期限は相談）
> - 中野：顧客 X 社から追加要件のヒアリング、次回定例で共有
> - 合意事項：リリースは 5/20 で確定
> - 検討事項：モバイル対応の優先度は次四半期に再検討

### 出力例

```
# 会議タスク一覧

## 会議情報
- 日時: 2026-04-23
- 参加者: 山田（PM）、佐藤（エンジニア）、鈴木（デザイナー）、中野（営業）
- 主要議題: 新機能 A 進捗、API 遅延、モバイル対応

## サマリー
- タスク計: 4 件（うち要確認 1 件）
- 決定事項: 2 件

## タスク一覧
| # | 担当者 | 内容 | 期限 | 報告先 | 優先度 |
| - | ------ | ---- | ---- | ------ | ------ |
| 1 | 山田 | 新機能 A 仕様書 v2 更新 | 2026-04-27（月） | Slack #dev | 高 |
| 2 | 佐藤 | API レスポンス遅延の原因調査・報告 | 2026-05-07 頃（GW 明け、要確認） | Slack #dev | 高 |
| 3 | 鈴木 | ログイン画面デザイン案検討 | 要確認（「相談」と発言のみ） | 次回定例 | 中 |
| 4 | 中野 | X 社から追加要件ヒアリング | 次回定例まで | 次回定例 | 中 |

## 決定事項
- リリース日は 2026-05-20 で確定
- モバイル対応の優先度は次四半期に再検討

## 要確認項目
- タスク #2: 「GW 明け」の具体日を佐藤さんに確認
- タスク #3: 期限が未確定（「相談」との発言のみ）
```
